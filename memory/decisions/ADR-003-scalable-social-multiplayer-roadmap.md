# ADR-003: 社交与多人联机的规模推广路线

**状态:** Accepted
**日期:** 2026-07-22
**决策人:** 项目 owner / coordinator

## 决策摘要

AI Werewolf 采用渐进式云架构：继续使用 Netlify 托管 Web 前端和短请求函数，继续使用 Supabase 托管认证、Postgres 数据库、RLS 和实时能力。第一版社交系统不维护自建 VPS 或本地生产服务器；第一版多人对局优先采用 Supabase Realtime + 服务端函数的方案。只有真实负载证明当前方案成为瓶颈时，才引入独立的长期运行游戏服务器。

当前产品阶段是：**玩家单机与 AI 对局，同时支持真人之间加好友、发消息和社交互动，但暂不支持真人联机对局。** 后续任务必须以此为当前阶段基线，不得把尚未实现的多人联机描述为现有能力。

## 背景与目标

产品长期目标是让玩家能够：

- 创建账号和个人资料；
- 添加好友、接受或拒绝好友申请、拉黑和举报；
- 在公共大厅发消息并看到在线状态；
- 创建队伍、邀请好友、加入或退出队伍；
- 进入多人 AI 狼人杀房间，和真人共同完成对局；
- 断线后恢复房间状态，并在对局结束后保存战绩。

这组能力同时包含持久数据、实时通信和游戏规则执行。三者不能被简单等同为“数据库”。持久数据可以由托管数据库负责，实时通信可以由托管 Realtime 能力负责，而游戏规则必须由可信的服务端执行，尤其是涉及隐藏身份、投票和 AI 行动时。

## 分阶段路线

### 阶段一：单机 AI 对局 + 社交基础（当前阶段）

玩家在浏览器中与 AI 完成狼人杀。真人之间可以登录、建立资料、加好友、发大厅消息、查看在线状态，并进行基础互动，但不能共同进入同一局游戏。

技术重点：

- 保持 Netlify 静态前端和函数部署方式；
- 使用 Supabase Auth、Postgres 和 RLS 建立社交数据；
- 使用 Supabase Realtime 实现大厅消息和在线状态；
- 继续让游客模式可用，游客数据只保存在本地；
- 不为多人游戏提前引入自建 WebSocket 服务；
- 不改变当前已验证的单机规则和 AI 体验。

阶段完成标准：好友和消息功能可用，未登录用户权限清晰，滥用入口有基本限制，生产数据库有 RLS 和备份策略，单机游戏无回归。

### 阶段二：小范围真人组队与多人 AI 房间

先开放受控的多人房间，而不是一次性实现完整社交平台。玩家可以创建房间、邀请好友、占座、准备、开始游戏、断线重连和结束后查看战绩。

首选技术方案：

- Supabase Postgres 保存房间、座位、完整对局状态和事件记录；
- Supabase Realtime 推送公开事件、成员变化和房间状态；
- Edge Function 或受保护的后端函数处理加入房间、准备、投票、技能使用等命令；
- AI 行动由服务端触发，API key 永不进入浏览器；
- 客户端只提交命令，不提交最终状态。

阶段完成标准：同一房间内多人看到一致的公开状态；每名玩家只能收到自己有权看到的秘密信息；重复请求、过期请求和断线重连不会破坏对局；异常房间可以恢复或安全结束。

### 阶段三：公开多人大厅与产品化运营

在小范围多人房间稳定后，开放更大范围的匹配和社交运营能力：公开房间、快速加入、好友邀请、房间筛选、举报、禁言、封禁、消息审核、战绩和基础反作弊。

技术重点：

- 房间生命周期和超时清理；
- 聊天消息限流、长度限制、敏感内容处理和举报链路；
- 连接断开、重连、浏览器刷新和重复点击的幂等处理；
- 关键游戏操作的审计事件；
- 错误监控、性能指标、数据库备份恢复演练；
- 按功能开关逐步放量，而不是一次性对全体用户开放。

阶段完成标准：可以观察主要业务指标和异常；用户能得到明确的错误和恢复路径；运营可以处理骚扰、刷屏、恶意退出和违规账号。

### 阶段四：按真实负载拆分实时游戏服务

当 Supabase Realtime、Edge Functions 或数据库出现持续性瓶颈时，再独立部署长期运行的游戏服务。该服务负责房间内的连接、回合调度和权威状态；Supabase 仍然负责账号、持久数据和最终战绩。

拆分触发条件应来自指标，而不是预先猜测用户规模。包括：

- 长期运行的活跃房间或连接数接近托管服务限制；
- 游戏命令延迟、Realtime 延迟或数据库写入延迟持续恶化；
- AI 行动队列持续积压，影响回合体验；
- 数据库读写压力与实时广播相互影响；
- 需要更细粒度的房间调度、区域部署或自定义扩容策略。

届时可选择托管容器、应用平台或专用实时服务，但不应在没有负载证据时提前增加运维复杂度。

## 目标架构

```text
浏览器
  ├─ Netlify CDN：React/Vite 前端
  ├─ Netlify Functions：AI 代理和短请求接口
  └─ Supabase
       ├─ Auth：登录和会话
       ├─ Postgres：资料、好友、消息、队伍、房间、战绩
       ├─ RLS：按用户和房间控制数据访问
       ├─ Realtime：大厅、在线状态和房间事件
       └─ Edge Functions：服务端业务命令和多人游戏入口

未来按需增加：
  ├─ 独立游戏服务器：长连接、房间调度、权威回合状态
  ├─ 队列/缓存：AI 行动、通知和削峰
  └─ 监控与日志：性能、错误、审计和运营指标
```

## 必须保持的架构不变量

### 1. 服务端权威

单机模式可以在客户端运行规则；多人模式不能信任客户端提交的最终状态。服务端必须校验阶段、座位、角色权限、目标、时间窗口和状态版本，并负责提交状态变化。

### 2. 秘密信息分层

完整对局状态只能保存在服务端。公开事件、玩家个人秘密信息和管理员信息必须分别生成，不能通过一次广播把其他玩家身份、查验结果或狼队信息泄露给客户端。

### 3. 规则与表达分离

`gameEngine.ts`、`beliefTracker` 和 `actionSelector` 继续承担规则、信念和行动决策。LLM 只负责表达，不得成为多人房间的规则裁判。多人模式应复用这些纯逻辑，并把执行位置逐步迁移到可信服务端。

### 4. 密钥只在服务端

模型 API key、服务端 Supabase key 和管理权限不能进入前端、源码仓库、日志或共享记忆。浏览器只使用公开客户端配置和受 RLS 保护的权限。

### 5. 所有权和访问控制先于功能开放

涉及好友、消息、队伍和房间的表必须同时设计用户身份、RLS、删除/拉黑语义和审计字段。不能先开放读写，再事后补权限。

### 6. 实时状态与持久数据分开思考

在线状态、正在输入和临时房间事件可以是短生命周期实时数据；好友关系、消息记录、房间结果和战绩必须持久化。不能把 Realtime 广播本身当作可靠数据库。

### 7. 可恢复和幂等

加入房间、发送邀请、投票、使用技能和结束对局等命令必须能够处理重复提交、网络重试、过期客户端和断线重连。多人房间需要状态版本或事件序列，不能只依赖客户端时间。

## 未来任务的范围判断规则

以后创建任务卡或修改代码时，先标注它属于哪个阶段：

- 阶段一任务：可以实现好友、资料、消息、在线状态和社交 UI，但不能假设真人已经能进入同一局；
- 阶段二任务：必须包含服务端房间状态、权限边界、重连策略和最小验证；
- 阶段三任务：必须同时考虑审核、举报、限流、封禁和运营可观测性；
- 阶段四任务：必须有真实负载或性能数据支持拆分基础设施。

任何任务在实现前都应回答：数据由谁拥有、谁能读取、谁能修改、秘密信息如何隔离、失败后如何恢复。只改前端状态而没有后端权限设计的多人功能，不视为完整实现。

## 主要风险与应对

| 风险 | 应对 |
| --- | --- |
| 客户端泄露隐藏身份 | 服务端保存完整状态，按玩家生成状态投影 |
| 聊天刷屏和骚扰 | 限流、长度限制、拉黑、举报、禁言和审核 |
| 断线造成房间卡死 | 心跳/超时、重连、状态版本和房间清理 |
| AI 请求变慢或失败 | 服务端队列、超时、重试、可解释的回退发言 |
| 第三方服务故障 | 数据备份、迁移脚本、服务边界和降级策略 |
| 过早自建基础设施 | 以延迟、错误率、连接数和队列积压作为拆分依据 |
| 青少年用户安全问题 | 最小化公开资料、举报封禁、隐私策略和运营审计 |

## 本 ADR 的执行结论

在当前阶段，继续完善单机 AI 游戏质量和社交基础设施。不要提前实现完整真人联机，也不要提前维护独立生产服务器。下一阶段只有在阶段一的好友、消息、资料、在线状态和权限基础稳定后，才设计多人房间协议和服务端权威状态。

任何与本路线冲突的任务，应在任务卡中明确说明冲突、理由和迁移成本，并由 owner/coordinator 决定是否修改本 ADR。
